# AI Request Classifier (Netpeak)

Сервіс для автоматичної класифікації та структурування вхідних запитів від внутрішніх команд AI-юніту.  
Читає "інбокс" з CSV → проганяє через LLM → видає структурований JSON + звіт + опціонально відправляє в Telegram/Google Sheets.

## 🚀 Як запустити

### 1. Клонувати та встановити залежності

```bash
git clone <repo-url>
cd netpeak-ai-classifier
python -m venv .venv
source .venv/bin/activate  # Linux/Mac
# .venv\Scripts\activate   # Windows
pip install -r requirements.txt
```

### 2. Налаштувати оточення

```bash
cp .env.example .env
```

Відредагувати `.env`:
- **Обов'язково:** `GEMINI_API_KEY` — отримати безкоштовно на https://aistudio.google.com/apikey
- **Опціонально:** Telegram-токени, Google Sheets credentials

### 3. Запуск

**Локально:**
```bash
python src/main.py
```

**Через Docker:**
```bash
docker-compose up --build
```

### 4. Результати

Після виконання в директорії `output/` з'являться:
- `output.json` — повний структурований результат
- `report.md` — агрегований звіт

## 🧠 Архітектура та ключові рішення

### Чому Gemini 2.0 Flash?
- **Безкоштовний tier** — не треба просити ключі у компанії
- **OpenAI-сумісний API** — працює через стандартний `openai`-клієнт + `instructor`
- **Швидкий** — 18 запитів обробляються за ~15-20 секунд
- **Підтримує tool calling** — дає стабільний структурований вивід

### Чому `instructor` + `pydantic`?
- Автоматична валідація відповіді LLM у строгу схему
- При невалідному виводі `instructor` **автоматично ретраїть** запит з поясненням помилки
- Типобезпека на всіх рівнях

### Чому async + `Semaphore`?
- Паралельна обробка прискорює роботу в ~3-5 разів
- `Semaphore(3)` захищає від `429 Too Many Requests` від безкоштовного Gemini
- Значення `SEM_LIMIT` конфігурується через `.env`

### Розширення схеми (чому додано)
До обов'язкових полів з ТЗ додано:
- `suggested_language` — мова запиту (uk/ru/en). Корисно для маршрутизації на різних моделів/промптів.
- `reasoning` — пояснення логіки класифікації. Критично для дебагу: якщо модель помилилась, одразу видно чому.

## ⚠️ Обмеження та де рішення може ламатися

### 1. Невалідний вивід LLM
**Що може статися:** модель повертає JSON, що не відповідає схемі (невірний enum, відсутнє поле).  
**Як закрито:** `instructor` автоматично ретраїть до 2 разів. Якщо все одно невалідно — запит потрапляє в `except`, логується, і фінальний список фільтрує `None`. У звіті це відображається як "кількість помилок".

### 2. Великий обсяг даних (1000+ запитів)
**Що може статися:**
- Rate limit від API
- Вичерпання пам'яті при зберіганні всіх результатів
- Довге виконання

**Як частково закрито:**
- `Semaphore` контролює паралельність
- `asyncio.gather` не блокує пам'ять

**Що не закрито (треба робити):**
- Чанкінг батчів (обробляти по 100 запитів з паузою)
- Запис у БД/файл по мірі обробки, а не в кінці
- Persistent queue (Celery/RQ) для продакшену

### 3. Недетермінізм LLM
**Що може статися:** один і той самий запит при двох запусках отримає різну категорію/пріоритет.  
**Як закрито:** `temperature=0.1` мінімізує рандом. Проте повністю детермінізму досягти неможливо.  
**Що зробив би далі:**
- Додати **few-shot examples** у промпт (2-3 еталонних приклади)
- Для спірних кейсів — **ансамбль з 3 запитів** і majority vote
- Кешування однакових запитів

### 4. Вартість токенів
**Поточна ситуація:** Gemini 2.0 Flash — безкоштовний (з лімітами). 18 запитів ≈ 10-15к токенів ≈ $0.  
**При масштабуванні:** для 1000 запитів/день на OpenAI GPT-4o-mini це ~$0.5-1/день, на GPT-4o — ~$5-10/день.  
**Оптимізації:**
- Кешування однакових запитів (хеш тексту → результат)
- Використання дешевших моделей для простих кейсів (наприклад, Haiku)
- Батчинг через batch API (OpenAI дає 50% знижку)

### 5. Довгі тексти
**Що може статися:** запит перевищує контекстне вікно.  
**Як закрито:** для поточних даних це не проблема (всі тексти короткі).  
**Що зробив би далі:**
- Перевірка довжини тексту перед відправкою
- Truncation або chunking для довгих запитів
- Окремий пайплайн для довгих текстів (спочатку саммарі, потім класифікація)

### 6. Конфіденційність даних
**Що може статися:** у `raw_text` можуть бути PII, паролі, токени.  
**Як закрито:** ніяк (це MVP).  
**Що зробив би далі:**
- PII-detector (наприклад, Microsoft Presidio) перед відправкою в LLM
- Анонімізація email, телефонів, IBAN
- On-premise LLM для чутливих даних

## 🔮 Що зробив би далі, якби було більше часу

1. **Few-shot prompting** — додати 3-5 еталонних прикладів у промпт для стабільнішої класифікації.
2. **Eval-пайплайн** — ручна розмітка 50 запитів як ground truth, автоматичний підрахунок accuracy/precision/recall.
3. **Web UI** — Streamlit-додаток для ручного перегляду та коригування класифікацій (active learning).
4. **База даних** — PostgreSQL замість JSON для запитів, історія змін, фільтрація.
5. **Webhook-інтеграція** — слухати Slack/Telegram напряму, без CSV-експорту.
6. **Моніторинг** — Prometheus + Grafana для метрик: час відповіді, вартість, кількість помилок.
7. **A/B тестування промптів** — фреймворк для порівняння різних версій промптів.
8. **Кешування** — Redis для однакових запитів.
9. **Human-in-the-loop** — запити з низькою впевненістю моделі йдуть на ручну перевірку.
10. **Fine-tuning** — якщо набрати 500+ розмічених запитів, дофінтюнити малу модель (наприклад, Llama 3.1 8B) для дешевшої та швидшої інференції.
